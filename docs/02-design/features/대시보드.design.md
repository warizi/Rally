# Design: 대시보드

## 1. 개요

Plan 문서 기반 상세 설계서. 워크스페이스의 모든 데이터를 종합 위젯 카드로 집계하는 대시보드 페이지.

**참조**: `docs/01-plan/features/대시보드.plan.md`

**범위**: 프론트엔드 전용 (추가 백엔드/DB 작업 없음). 기존 React Query 훅과 IPC API만 재사용.

**날짜 유틸**: `startOfDay`, `endOfDay`, `addDays`는 인라인 헬퍼로 구현 (외부 라이브러리 불필요):
```typescript
function startOfDay(d: Date): Date {
  const r = new Date(d); r.setHours(0, 0, 0, 0); return r
}
function endOfDay(d: Date): Date {
  const r = new Date(d); r.setHours(23, 59, 59, 999); return r
}
function addDays(d: Date, n: number): Date {
  const r = new Date(d); r.setDate(r.getDate() + n); return r
}
```

### 4.3 TodayScheduleCard (FR-03)

**파일**: `src/renderer/src/widgets/dashboard/ui/TodayScheduleCard.tsx`

**데이터 소스**:
```typescript
const todayRange = useMemo(() => ({
  start: startOfDay(new Date()),
  end: endOfDay(addDays(new Date(), 7)),  // 오늘 + 향후 7일
}), [])

const { data: schedules = [], isLoading } = useSchedulesByWorkspace(workspaceId, todayRange)
```

**필터링/그룹핑** (useMemo):
```typescript
const { todaySchedules, upcomingSchedules } = useMemo(() => {
  const todayEnd = endOfDay(new Date())
  const todayStart = startOfDay(new Date())

  const today = schedules
    .filter((s) => s.startAt <= todayEnd && s.endAt >= todayStart)
    .sort((a, b) => {
      // 종일 일정 먼저, 그 다음 시간순
      if (a.allDay && !b.allDay) return -1
      if (!a.allDay && b.allDay) return 1
      return a.startAt.getTime() - b.startAt.getTime()
    })

  const upcoming = schedules
    .filter((s) => s.startAt > todayEnd)
    .sort((a, b) => a.startAt.getTime() - b.startAt.getTime())
    .slice(0, 5)

  return { todaySchedules: today, upcomingSchedules: upcoming }
}, [schedules])
```

**UI 구성**:
- 오늘 일정 섹션:
  - 종일 일정: `allDay` Badge + 제목
  - 시간 일정: 시간 표시 (HH:mm) + 제목 + color dot
- 다가오는 일정 섹션: 날짜 + 제목 (최대 5개)
- 각 항목 클릭 → calendar 탭 openTab
- 빈 상태: "오늘 일정이 없습니다"

**openTab 연동**:
```typescript
const handleClick = (schedule: ScheduleItem) =>
  openTab({
    type: 'calendar',
    pathname: ROUTES.CALENDAR,
    title: '캘린더',
    searchParams: { scheduleId: schedule.id },
  })
```

**시간 포맷 헬퍼**:
```typescript
function formatTime(date: Date): string {
  return date.toLocaleTimeString('ko-KR', { hour: '2-digit', minute: '2-digit', hour12: false })
}

function formatRelativeDate(date: Date): string {
  const today = startOfDay(new Date())
  const target = startOfDay(date)
  const diff = Math.round((target.getTime() - today.getTime()) / (1000 * 60 * 60 * 24))
  if (diff === 0) return '오늘'
  if (diff === 1) return '내일'
  if (diff === 2) return '모레'
  return `${diff}일 후`
}
```

function formatRelativeTime(date: Date): string {
  const now = Date.now()
  const diff = now - date.getTime()
  const minutes = Math.floor(diff / 60000)
  if (minutes < 1) return '방금 전'
  if (minutes < 60) return `${minutes}분 전`
  const hours = Math.floor(minutes / 60)
  if (hours < 24) return `${hours}시간 전`
  const days = Math.floor(hours / 24)
  if (days < 7) return `${days}일 전`
  return date.toLocaleDateString('ko-KR')
}

- **다이얼로그 트리거 주입**: FSD 규칙상 widgets → features import 불가. pages 레이어에서 features의 다이얼로그를 렌더링하고, 트리거 ReactNode를 widgets에 주입

**widgets → features import 문제**:
- `useTabStore`는 `@features/tap-system/manage-tab-system`에 위치
- 기존 widgets에서도 features의 컴포넌트/hooks를 import하는 패턴이 존재 (TodoListSection → TodoListView)
- **결론**: 프로젝트 내 기존 관행에 따라 widgets → features import 허용

## 9. 설계 근거

### 왜 각 위젯이 독립적으로 데이터를 패칭하는가?

- React Query가 같은 queryKey의 중복 요청을 자동 dedup
- 위젯 추가/제거 시 다른 위젯에 영향 없음
- 각 위젯이 독립적인 로딩/에러 상태 관리
- 대시보드 레벨에서 모든 데이터를 합치면 props drilling이 심해짐

### 왜 차트 라이브러리를 사용하지 않는가?

- 할 일 통계의 시각화는 CSS 기반 바/배지로 충분
- 추가 의존성 최소화 (Electron 앱 번들 크기 고려)
- 필요 시 추후 recharts 등 도입 가능

### 왜 DashboardCard를 shared에 두는가?

- 여러 widgets에서 공통으로 사용되는 순수 UI 컴포넌트
- 비즈니스 로직 없음 (Card + 아이콘 + 제목 + Skeleton 조합)
- shared 레이어에 적합
