# Rally 할일 관리 방법론

Rally 워크스페이스에서 할일을 생성하고 구조화할 때 따르는 원칙.

## 언제 적용하는가

- `mcp__rally__manage_todos`로 할일을 생성/수정할 때
- 기능을 할일로 분해할 때

## 할일 생성 원칙

### 1. 제목은 완료 조건이 명확하게

```
좋음: "터미널 패널 Store 구현"
나쁨: "터미널 관련 작업"
좋음: "워크스페이스 전환 시 터미널 destroy 처리"
나쁨: "터미널 버그 수정"
```

"~한다", "~구현" 형태로 완료 시점이 명확한 제목을 쓴다.

### 2. 상위-하위 구조

기능 단위로 상위 할일 1개, 실행 단위로 하위 할일 N개:

```
상위: PDF 뷰어 기능 구현
  ├── 하위: PDF 파서 라이브러리 선정 및 설치
  ├── 하위: PDF 렌더링 컴포넌트 구현
  ├── 하위: 탭에서 PDF 열기 연동
  └── 하위: 확대/축소/페이지 이동 UI
```

### 3. 분해 기준

하위 할일 1개 = **하나의 작업 세션에서 끝낼 수 있는 크기**

분해 가이드:

- **3~7개**가 적당 (2개 이하면 분해 불필요, 8개 넘으면 중간 그룹 고려)
- 레이어별로 분리: DB → 서비스 → UI → 연동
- 각 하위 할일이 독립적으로 완료 가능해야 한다
- 의존 순서가 있으면 제목에 번호를 붙인다: "1. 스키마 정의", "2. 서비스 구현"

### 4. subtodos 활용

`manage_todos`의 `subtodos` 배열로 상위와 하위를 한 번에 생성한다:

```json
{
  "action": "create",
  "title": "PDF 뷰어 기능 구현",
  "priority": "high",
  "subtodos": [
    { "title": "1. PDF 파서 라이브러리 선정" },
    { "title": "2. PDF 렌더링 컴포넌트 구현" },
    { "title": "3. 탭 연동" }
  ]
}
```

## 우선순위 기준

| priority | 기준                                            |
| -------- | ----------------------------------------------- |
| high     | 다른 작업이 이것에 의존하거나, 마감이 임박한 것 |
| medium   | 일반적인 기능 개발, 개선                        |
| low      | 있으면 좋지만 급하지 않은 것                    |

## 상태 전이

```
할일 → 진행중 → 완료
  ↓               ↑
보류 ─────────────┘
```

- **할일**: 아직 착수하지 않은 상태
- **진행중**: 현재 작업 중 (한 번에 1~2개만 진행중 권장)
- **완료**: 작업이 끝나고 검증된 상태
- **보류**: 외부 의존, 우선순위 변경 등으로 일시 중단

## linkedItems 활용

### 할일 생성 시

- 관련 노트가 있으면 `linkItems`로 연결한다
- 관련 캔버스가 있으면 함께 연결한다
- 하위 할일(subtodo)에는 linkItems를 걸 수 없으므로 **상위 할일에 연결**한다

### 할일 완료 시

- 구현 결과를 노트로 정리했다면 해당 노트도 연결한다
- 관련 할일이 모두 완료되면 상위 할일도 완료로 변경한다

## 배치 처리

여러 할일을 한 번에 생성/수정할 때는 `actions` 배열로 배치 처리한다.
개별 API 호출보다 효율적이다.

```json
{
  "actions": [
    { "action": "create", "title": "기능 A", "priority": "high" },
    { "action": "update", "id": "xxx", "status": "완료" },
    { "action": "update", "id": "yyy", "status": "진행중" }
  ]
}
```
