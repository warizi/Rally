# Rally 캔버스 시각화 방법론

Rally 캔버스를 생성할 때 따르는 레이아웃, 색상, 구성 원칙.

## 언제 적용하는가

- `mcp__rally__create_canvas`로 새 캔버스를 생성할 때
- `mcp__rally__edit_canvas`로 기존 캔버스를 수정할 때

## 노드 배치 원칙

### 1. 흐름 방향을 통일한다

- **기본**: 위→아래 (top-down) — 계층 구조, 실행 순서
- **대안**: 좌→우 (left-to-right) — 타임라인, 파이프라인
- 한 캔버스에서 방향을 섞지 않는다

### 2. 그리드 정렬

노드 좌표를 일정 간격으로 배치한다:
- **수직 간격**: 노드 height + 80~120px
- **수평 간격**: 노드 width + 60~100px
- 같은 계층의 노드는 같은 y좌표에 배치한다

```
y=0     [  개요  ]
y=400   [ 상세A ] [ 상세B ] [ 상세C ]
y=800   [ 결론  ]
```

### 3. 참조 노드는 크게

`note`, `csv` 타입 참조 노드(refId 사용)는 내용이 보여야 하므로 충분히 크게:
- **최소**: width 400, height 300
- **권장**: width 500, height 400 (내용이 많은 노트)
- **테이블**: width 600, height 350

`text` 타입 노드는 내용에 맞게 조절:
- 짧은 라벨: width 200, height 80
- 설명 텍스트: width 300, height 150

## 색상 체계

### 역할별 색상 규칙

| 역할 | 색상 | 용도 |
|------|------|------|
| 핵심/개요 | 보라 (`#a78bfa`) | 메인 노트, 중심 개념 |
| 실행/할일 | 파랑 (`#60a5fa`) | 할일, 작업 단위 |
| 주의/경고 | 빨강 (`#f87171`) | 주의사항, 제약조건 |
| 완료/확인 | 초록 (`#4ade80`) | 완료된 항목, 검증된 사항 |
| 보조/참고 | 회색 (`#9ca3af`) | 부가 정보, 참고 자료 |
| 데이터 | 주황 (`#fb923c`) | 테이블, 스키마, 데이터 |

한 캔버스에 3~4가지 색상만 사용한다. 너무 많으면 의미가 희석된다.

## 엣지(연결선) 원칙

### 1. 의존/순서 관계만 연결

모든 노드를 연결하지 않는다. 의미 있는 관계만:
- 실행 순서 (1→2→3)
- 의존 관계 (A가 있어야 B 가능)
- 참조 관계 (기획 노트 → 구현 할일)

### 2. 라벨 활용

관계가 명확하지 않은 엣지에는 라벨을 붙인다:
- "참조", "의존", "생성", "호출"

## 캔버스 구성 패턴

### 아키텍처 맵

```
[전체 개요 노트]
    ↓
[모듈A] [모듈B] [모듈C]
    ↓       ↓       ↓
[상세A] [상세B] [상세C]
```

### 기획 보드

```
[기획 노트] ──→ [할일1]
                [할일2]
                [할일3]
[주의사항] ────→ [할일2]
```

### 흐름도

```
[시작] → [단계1] → [단계2] → [완료]
                ↘ [예외처리] ↗
```

## description 작성

캔버스 생성 시 description에 캔버스의 목적을 한줄로 적는다:
- 좋음: "터미널 기능의 탭→패널 전환 구현 구조 시각화"
- 나쁨: "터미널"
